Introducción del Proyecto

En el entorno académico y profesional actual, la incompatibilidad de formatos multimedia representa un obstáculo diario. VIDEOFORGE 2.0 nace como una solución de escritorio robusta en C# que encapsula comandos complejos de transcodificación mediante el motor Open Source de FFmpeg 7.x, eliminando la dependencia de wrappers ineficientes y ofreciendo control a nivel del sistema operativo.

Gestión de Formatos

Conversiones cruzadas fluidas entre extensiones líderes: MP4, MKV, AVI, MOV, FLV, WMV y WebM sin pérdida de calidad.

Control Concurrente

Procesamiento paralelo asíncrono regulado por semáforos lógicos para optimizar el consumo de la CPU del usuario.

Mantenibilidad MVC

Arquitectura formal que separa estrictamente la visualización (UI), los modelos de datos y la lógica del convertidor.

Problemática vs Solución

La Problemática

  • Incompatibilidad multimedia: Dificultad recurrente de estudiantes y docentes en el uso de plataformas virtuales de la UNAJ.
  • Bloqueo del sistema: Los convertidores comunes saturan los núcleos de la CPU, provocando el congelamiento completo del SO.
  • Cero aceleración: Desaprovechamiento de las capacidades de aceleración gráfica por GPU disponibles en las computadoras.
  • Pérdida de avances: Ante conversiones canceladas, los archivos intermedios quedan corruptos o consumiendo espacio en disco.

Nuestra Solución

  • Transcodificación asíncrona: Procesamiento desacoplado de la interfaz gráfica usando tareas en segundo plano.
  • Gestión de concurrencia: Implementación de semáforos para limitar dinámicamente las conversiones paralelas.
  • Detección de GPU (WMI): Selección autónoma de codecs acelerados (NVENC, AMF, QSV) según el fabricante.
  • Control nativo del proceso: Capacidad de pausar/reanudar el flujo suspendiendo los hilos a nivel del kernel de Windows.

Requisitos No Funcionales (RNF)

Rendimiento (RNF-01)

Tasa de Procesamiento

Conversión menor a 2 minutos por 1 GB de video usando CPU pura, y menor a 45 segundos por 1 GB al habilitar aceleración por hardware GPU.

Usabilidad (RNF-02)

Curva de Aprendizaje

La interfaz gráfica intuitiva tipo Dashboard debe asegurar que un usuario común aprenda a operar las funciones de conversión en menos de 10 minutos sin entrenamiento previo.

Seguridad (RNF-03)

Protección de Accesos

Las contraseñas se almacenan obligatoriamente con hash SHA-256 en base de datos. Las sesiones se restringen mediante perfiles de acceso (Nivel 1: Administrador, Nivel 2: Estándar).

Confiabilidad (RNF-04)

Tolerancia a Fallos

Cobertura del 100% de bloques estructurados `try-catch`. Si ocurre un error o se cancela el proceso, se ejecuta la remoción automática de fragmentos de video corruptos en disco.

Escalabilidad (RNF-05)

Multitarea Controlada

Límite configurable de hilos lógicos en la cola. La transcodificación masiva no debe elevar el uso de CPU a niveles de inestabilidad general del sistema operativo anfitrión.

Portabilidad (RNF-08)

Configuración Externa

Toda la cadena de conexión de MySQL es editable mediante un archivo de configuración `.ini` externo. El sistema se puede redirigir a un servidor diferente sin necesidad de recompilar.

Gestión de Concurrencia

Patrón Productor-Consumidor

Para evitar la saturación de los recursos lógicos de la computadora de escritorio y el consecuente congelamiento de la interfaz de usuario, se implementa una arquitectura basada en tareas asíncronas y semáforos lógicos:

  • SemaphoreSlim: Instanciado como `new SemaphoreSlim(maxHilos)` donde `maxHilos` es definido por el usuario o sugerido según los núcleos lógicos detectados.
  • Await asíncrono: Las tareas en cola ejecutan `await semaforo.WaitAsync()` liberando el hilo principal del procesador.
  • Thread-Safety: Las actualizaciones del DataGridView del progreso se enrutan mediante `Invoke` o `InvokeRequired` para asegurar la integridad de la UI desde hilos secundarios.

Lógica en C# (Form_converter.cs)

// Control dinámico de conversiones simultáneas
private async Task ProcesarColaConversiones() {
  List<Task> tareas = new List<Task>();
  foreach (var video in listaVideos) {
    await semaforo.WaitAsync();
    tareas.Add(Task.Run(async () => {
      try {
        await video.Convertir(cts.Token);
      } finally {
        semaforo.Release();
      }
    }));
  }
  await Task.WhenAll(tareas);
}

Integración FFmpeg + GPU

Detección Inteligente de Hardware

El motor analiza la controladora gráfica principal mediante consultas WMI (`Win32_VideoController`). Si se detecta hardware dedicado compatible, se inyecta el códec optimizado en los argumentos de FFmpeg:

  • NVIDIA (h264_nvenc): Ejecuta transcodificación directa por hardware utilizando núcleos CUDA y tasa de calidad controlada (`-cq 20`).
  • AMD (h264_amf): Invoca el Advanced Media Framework para acelerar el procesamiento.
  • Intel (h264_qsv): Habilita Quick Sync Video para CPUs Intel compatibles.
  • CPU (libx264): Algoritmo de respaldo de alta calidad (`-crf 20`) si no existe GPU compatible activa.

Control Nativo de Subprocesos

La pausa y reanudación no destruyen el subproceso del ejecutable CLI, se suspenden sus hilos lógicos en la memoria RAM llamando al Kernel de Windows: 

// Importación de APIs nativas de Windows (ntdll.dll)
[DllImport("ntdll.dll")]
public static extern int NtSuspendProcess(IntPtr processHandle);

[DllImport("ntdll.dll")]
public static extern int NtResumeProcess(IntPtr processHandle);

Extracción de Progreso: Mediante expresiones regulares sobre el stream `stderr` de FFmpeg: `Duration:\s*(\d+):(\d+):(\d+)` y `time=(\d+):(\d+):(\d+)`.

Seguridad y Persistencia BD

Seguridad en Capas

  • SHA-256 Hashing: Las contraseñas se convierten a hash de 64 caracteres hexadecimales antes de guardarse o validarse. Nunca se almacenan ni viajan en texto plano.
  • Stored Procedures: Toda la autenticación reside dentro del motor de MySQL mediante el procedimiento `Acceder(Usuario, Clave)`. Evita vulnerabilidades de Inyección SQL.
  • Sesión Estática: Los datos del usuario logueado quedan protegidos en la clase estática en memoria `Sesion`, restringiendo dinámicamente los menús según el rol de acceso.

Base de Datos MySQL (db_dinamic)

-- Procedimiento seguro de Login
CREATE PROCEDURE Acceder(IN Usr VARCHAR(50), IN Pwd VARCHAR(200))
BEGIN
  SELECT usersId, grupoId, nombres, nivel 
  FROM usuarios 
  WHERE users = Usr AND clave = SHA2(Pwd, 256) AND estado = 1;
END;

Estructura Relacional: Entidad principal `usuarios` con llave foránea a `grupos` para definir validez temporal de acceso e historiales de auditoría de conversión.

Diagrama de Secuencia

Traza paso a paso la transcodificación de archivos por lotes: El formulario captura el directorio, levanta los metadatos con `FFprobe`, inicializa el semáforo concurrente e inicia hilos del motor FFmpeg secuencialmente.

Estructura formal de persistencia. Consiste en la tabla `usuarios` (usersId, grupoId, nombres, users, clave SHA-256, nivel de rol, email, fecha de creación) y la tabla `grupos` (grupoId, usersId, nombreGrupo, fechaInicio, fechaFinal) que limita el acceso según vigencia académica.

Modelo Dinámico Simplificado

Modelado del sistema bajo la metodología de Forrester (Stock & Flow):

Concepto Dinámico Elemento en VIDEOFORGE 2.0 Comportamiento y Regulación
Stock (Acumulador) Cantidad de archivos en cola de conversión (`listaVideos`). Acumula los videos pendientes de procesamiento seleccionados por el usuario.
Flujo de Entrada Tasa de carga de videos desde la carpeta origen. Incrementa el Stock inicial de videos pendientes en el sistema.
Flujo de Salida Tasa de conversión exitosa por minuto. Reduce el stock de videos pendientes al terminar el procesamiento de FFmpeg.
Regulador Control de hilos paralelos (`SemaphoreSlim`). Controla la velocidad del flujo de salida limitando los procesos activos concurrentes.
Auxiliar GPU Acelerador de tasa de salida. Incrementa la velocidad de salida si se dispone de aceleración de hardware activa.
Feedback Negativo Lazo de balance de carga. Mayor stock de videos en cola aumenta el tiempo de espera, forzando la cola de procesamiento.
